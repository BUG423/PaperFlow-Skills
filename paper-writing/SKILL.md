# paper-writing: 7 步逐段写完一篇论文

## 这个 skill 是什么

一台**线性流水线工具**,把 `paperflow/experiment-plan.md` + `paperflow/results.md` 转成 `paper/` 目录下完整的英文 LaTeX 论文 + `paperflow/claims.md` (给 prereview 的硬接口)。

**逐段交付**: 不是一次性输出整章,每段输出后等用户"通过/改 X/重写"再写下一段。

## Step 1: 读上游交接物

**强制**读以下文件 (找不到就走 standalone,但提示用户先跑前置 skill):
- `paperflow/experiment-plan.md` — claim/evidence 对账表 + 目标 venue
- `paperflow/results.md` — 实验事实来源 (**所有数字必须来自这里**)
- `paperflow/references.bib` — `\cite{}` 的 key 以这个为准
- `paperflow/ideation.md` (可选) — 看故事线背景
- `paperflow/review.md` (若存在) — 这是 prereview 之后的修改轮次,优先处理标 `回到 writing` 的"必须改"条目

**关键检查**: 如果 `results.md` 里还有 `[待填写]`,**明确告诉用户哪些表还空**,不替他编数字。**实验未跑完时可以先写方法/相关工作章节,但不能写实验章节**。

## Step 2: 故事线对齐 (≤ 1 轮)

**话术**:

> Step 1 完成: 读到 <venue> 投稿 + <N> 个核心 claim + 实验 <已跑完/部分完成>。
>
> **Step 2/7**: 对齐故事线。用一段话把整篇论文讲通,这段话是后续所有写作的"宪法"——摘要/引言/方法/实验/结论必须和它一致。
>
> 模板 (你帮用户填,空缺的让用户补):
>
> > 有一个 **[具体问题]**,现有的 **[某某方法路线]** 因为 **[某个根本原因]** 解决不了这个问题。我们观察到 **[某个关键现象]**,基于这个观察提出了 **[一个方法]**,它的核心思路是 **[主要机制]**。我们在 **[什么设定下]** 做了实验,主要结果是 **[最重要的数字或发现]**。这件事的意义在于 **[对领域有什么启发]**。

填完确认: 如果用户说"还行就这样",直接进 Step 3。如果用户说某句不准/不通,改到他点头为止。**故事线不通就别开始写——回 paper-ideation**。

## Step 3: 素材清单 + 写作顺序 (≤ 1 轮)

**3.1 素材清单**

```markdown
✅ 已有: 主表(Table 1) / 消融(Table 2) / 方法流程图(Fig 1)
⚠️ 缺: 和 Baseline X 的对比 (跑到 80%); 定性可视化 (没做)
❌ 没有计划: 效率对比表 (审稿人可能问)
```

对 ⚠️ ❌ 的部分**不替用户承诺会做**——问"这部分缺了审稿人会说什么? 要不要我帮你设计补充实验? 回 paper-experiment"。

**3.2 写作顺序 (推荐 vs 自然顺序)**

```markdown
推荐顺序 (高效):
1. 方法 (最独立,写完后引用有锚)
2. 实验 (有方法基础后写)
3. 引言 (此时故事熟透,写得精准)
4. 相关工作 (引言写完后映射 gap)
5. 摘要 (最后写,最精炼)
6. 结论 (简短收尾)
7. 标题 (最后敲定)

自然顺序 (用户偏好则用): 引言 → 相关工作 → 方法 → 实验 → 结论 → 摘要 → 标题
```

让用户选一个,记录 `<write_order>`。

## Step 4: 逐段写中文初稿 (主体)

每一段用固定 5 步节奏:

```
1. 【定位】这一段是 <章节> 第 <N> 段,功能是 <承上启下 / 提出 / 论证 / 总结>。前面 N-1 段建立了 X,接下来这段要做 Y。
2. 【正文】(中文段落,5-12 句)
3. 【注释】(可选) 某些措辞或结构选择的讲究,e.g. "我把 claim 从 'A 解决了 X' 改成 'A 针对 X 的关键瓶颈',因为前者会被审稿人觉得 overclaim"
4. 【预告】下一段要做什么 (一句话)
5. 【确认】这段可以吗? 要改什么? (回复 "通过" / "改 xxx" / "这段重写")
```

**写作纪律**:
- **不替用户做主张**: 实验只差 0.3% 不说"significant",问"统计检验支持吗? 措辞改为 competitive?"
- **写作不是翻译实验表**: 实验章节必须有叙事——哪组对比说明了什么 / Baseline A 为什么不行 / 异常值原因
- **每个数字必须来自 results.md**: 引用具体 table/figure,**禁止凭记忆填数字**

### 4.1 各章节心法

**方法**: 按 reviewer 理解的逻辑链写,不是代码执行顺序。先 overview → 最核心模块 (你的创新点) → 辅助模块 + 实现细节。

**实验**: 让数据讲故事,不要逐行念数字。每展示一个结果,紧接着解释"为什么"。

**引言**: 漏斗结构 — 宽泛领域 → 具体 gap → 你的方案 → 贡献 list。

**相关工作**: 按**论证线**分节而非按时间/作者分节。"使用 X 的方法但做法不同"、"解决 Y 但角度不同"、"借鉴 Z 但场景不同"。

### 4.2 修辞纪律 (一律改)

| 避免 | 替代 |
|------|------|
| "我们的方法达到了 SOTA 性能" | "我们的方法在 3 个 benchmark 上超过现有最佳" (跟具体数字) |
| "module A 显著提升了性能" | "加入 module A 带来了 1.2% 的提升 (Table 3)" |
| "大量实验证明" | "我们在 4 个数据集 / 3 种设定下做了验证 (Section 4.2-4.5)" |
| "方法具有很好的鲁棒性" | "在噪声水平 0-20% 范围内,性能下降不超过 2% (Figure 5)" |
| "我们相信方法可以推广到..." | "本工作在 X 上验证了有效性,其他模态/领域的适用性需进一步研究" |

## Step 5: 翻译为英文 LaTeX

中文初稿全部写完且确认后才进入这一步。同样**逐段翻译**。

**翻译原则**:
- 中文初稿的事实/数字/引用/逻辑结构不变
- 学术英文追求清晰 > 华丽: 短句 > 长句; 主动语态 > 被动 (方法部分被动是惯例)
- 术语全文一致 — 同一概念全文用同一英文词
- 中文里"显著"、"大量"、"极大"等强调词替换为具体数字或删除
- LaTeX 格式正确: `\section{}`, `\cite{}`, `\ref{}`, `\label{}`

**小提示**: 如果某句翻译特别别扭,可能中文原文逻辑就有问题——回中文阶段改,而不是硬翻。

## Step 6: 自我诊断 (写完后)

按下面 5 条扫一遍:

```markdown
1. 只看摘要 + 引言前两段 → 能说清贡献吗?
2. 只看图表标题 (不看正文) → 故事线还在吗?
3. 每个 section 的第一句和最后一句 → 逻辑连贯吗? 有断裂吗?
4. 引言里的每个 claim 在方法/实验里都有对应支撑吗?
5. 哪里 overclaim? 哪里暗示了证据没支持的东西?
```

任何一条 fail,**回 Step 4 改对应章节**——不靠"换几个词"掩盖。

## Step 7: 生成 claim 台账给 prereview

把 Step 6 第 4-5 条的结果落成 `paperflow/claims.md`:

```markdown
> stage: writing | updated: <YYYY-MM-DD>

# Claim 台账
| 论文中的 claim | 出现位置 (section) | 证据锚点 (table/fig/section) | 措辞强度 |
|----------------|-------------------|-----------------------------|----------|
| 在 A/B/C 上超过 SOTA | Abstract, §1, §4.1 | Table 1 | strong (已核: +1.2%) |
| 模块 M 是提升主因 | §4.3 | Table 3 消融 | medium |
| 方法对噪声鲁棒 | §4.5 | Figure 5 | weak (仅单一设定,标记待补) |
```

**逐条对照 `results.md`**: 凡是论文里写了但 `results.md` 不支撑的 claim,当场标红并提醒用户改措辞或补实验,**不要让它流到 prereview 才被发现**。

更新 `manifest.md` 的 `writing` 行为 `done`。

## 跨步骤纪律

- **一次只写一段**: 不是效率问题,是控制权问题。一次输出一章用户会"嗯还行"过掉,逐段逼用户审视
- **数字单一来源**: 全文每个数字必须能 trace 到 `results.md` 的具体 table/cell
- **review.md 反馈优先**: 如果存在 `review.md`,Step 1 后先列出标 "回到 writing" 的必改项,Step 4 逐条改完后在条目旁标记日期
- **不替用户做主张**: 任何"significant / SOTA / robust / general" 之类的强 claim 都要先问用户"数据支撑这个措辞强度吗"
